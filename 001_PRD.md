## 一、项目概述

在简历上放置二维码，HR扫码后进入手机端聊天界面（免登录），与AI机器人对话。机器人基于 **LLM-Wiki 知识库**（Obsidian Vault 中已整理的结构化 markdown），通过 RAG 检索 + LLM 生成，**准确回答简历相关问题**。

核心原则：**准确性是生命线。无法保证回复准确，二维码就是败笔。**

## 二、目标用户

| 角色 | 场景 |
|------|------|
| HR / 猎头 / 面试官 | 手机扫码，快速了解候选人背景、项目细节、技能匹配度 |
| 候选人（本人） | 维护 LLM-Wiki 知识库，保持数据新鲜度 |

## 四、知识库现状分析

### LLM-Wiki 数据概况

Wiki 位于 Obsidian Vault（SMB 挂载 `/mnt/obsidian_wiki`），采用 Karpathy LLM-Wiki 模式：

```
obsidian_wiki/
├── SCHEMA.md              # 结构约定
├── index.md               # 内容目录
├── log.md                 # 操作日志
├── 001_Raw/               # 原始资料（不可变）
├── 002_Entities/          # 实体页（人物、公司、产品）
├── 003_Concepts/          # 概念页
├── 004_Sources/           # 来源页（含简历原文）
└── 005_Synthesis/         # 综合分析
```

### 简历相关数据量

| 类别 | 文件 | 字符数 |
|------|------|--------|
| 个人主页 | 傅颉 Freddie.md | 5,419 |
| 工作经历 | 百融云创/自如/博洛尼/译禾 | 7,164 |
| 项目经历 | Bonjour!/Loekalization/SeeDAO | 3,470 |
| 简历原文 | Freelancer/HR-Resume (中英文) | 5,437 |
| 方法论 | STAR法则.md | 2,049 |
| 综合分析 | 职业发展路径与AI化趋势 | 8,458 |
| **合计** | **13个核心文件** | **~32,000 字符 ≈ 16,000 tokens** |

### 准确性保障策略

16K tokens 超出了"全部塞进 system prompt"的舒适区，必须用 **RAG**：

1. **检索阶段**：用户问题 → embedding → 从 46 个 wiki 页面中检索 top-K 相关页面
2. **上下文组装**：system prompt（角色约束）+ 检索到的 wiki 原文 + 用户问题
3. **生成阶段**：LLM 严格基于上下文生成回答，不确定时明确说明
4. **幻觉防护**：system prompt 约束"仅根据提供的知识库内容回答，不得编造"

## 五、系统架构

### 双节点架构

```
简历PDF → QR码 → askme.yourdomain.com
                      │
              ┌───────┴───────┐
              │  DNS / 负载均衡  │
              └───┬───────┬───┘
                  │       │
        ┌─────────┘       └─────────┐
        ▼                           ▼
  节点A：树莓派（主）          节点B：外网（兜底）
  ┌─────────────────┐      ┌─────────────────┐
  │ FRP 内网穿透     │      │ Cloudflare Pages │
  │ (已有，运行中)   │      │ + Workers        │
  │                 │      │                  │
  │ RPi Docker:     │      │ 静态前端 +       │
  │ ┌─────────────┐│      │ Worker API       │
  │ │ FastAPI      ││      │                  │
  │ │ + 聊天前端    ││      │ 向量索引(内嵌)   │
  │ │ + RAG引擎    ││      │                  │
  │ └──────┬──────┘│      └─────────────────┘
  │        │       │             ▲
  │  LLM-Wiki      │             │
  │  (SMB 实时读取) │      知识库快照(定期同步)
  └────────────────┘
        │                │
        └───── 共用 ─────┘
              LLM API (Z.AI GLM / DeepSeek)
```

### 节点职责

| | 节点A：树莓派（主） | 节点B：外网（兜底） |
|--|--|--|
| **访问方式** | FRP 隧道 → `media.logth.ink:端口` | Cloudflare Pages 域名 |
| **前端** | FastAPI 托管单页HTML | 静态部署 |
| **后端** | FastAPI Docker 容器 | Cloudflare Worker |
| **知识库** | 实时读取 SMB 挂载的 Obsidian Vault | 静态快照（定期同步） |
| **向量检索** | 本地 embedding 文件（JSON） | Worker 内嵌向量索引 |
| **LLM** | 调用 Z.AI GLM API | 调用同一 LLM API |
| **优势** | 数据实时、修改即生效 | 全球CDN、高可用 |
| **劣势** | 依赖家庭网络、RPi稳定性 | 数据非实时（延迟同步） |

### 故障切换逻辑

```
DNS 解析 askme.yourdomain.com
    │
    ├─→ 优先指向 FRP 隧道（树莓派）
    │   ├─ 响应正常 → 使用节点A
    │   └─ 超时/不可达 → 自动切换
    │
    └─→ 兜底指向 Cloudflare Pages（节点B）
        └─ 始终可用
```

实际方案：DNS 设置两条记录，前端页面做健康探测，节点A不可用时自动切换到节点B。或者更简单：QR码直接用 Cloudflare Pages 域名（保证始终可达），页面内部优先请求 FRP 隧道 API，失败则 fallback 到 Worker API。

## 七、技术选型

### 节点A（树莓派）

| 组件 | 技术 | 理由 |
|------|------|------|
| 运行时 | Python 3 + FastAPI | 轻量，RPi 友好 |
| 前端 | 单页 HTML + Tailwind CDN + Vanilla JS | 极简，无构建步骤 |
| RAG 引擎 | 自写检索器（embedding 预计算 + 余弦相似度） | 无需额外数据库 |
| Embedding 模型 | 在线 API（GLM embedding / bge-m3 via API） | RPi 跑不动本地模型 |
| 向量存储 | JSON 文件（~46个页面向量，<2MB） | 数据量极小，文件即可 |
| LLM | Z.AI GLM-5-turbo API | 已有订阅，中文好 |
| 容器化 | Docker + docker-compose | 已有 Docker 环境 |
| 知识库读取 | 直接读取 SMB 挂载的 `/mnt/obsidian_wiki/` | 实时，改即生效 |

### 节点B（外网兜底）

| 组件 | 技术 | 理由 |
|------|------|------|
| 前端 | Cloudflare Pages（同一份 HTML） | 全球CDN，免费 |
| 后端 | Cloudflare Worker | Serverless，免费 |
| RAG | Worker 内嵌向量索引（JSON 静态导入） | 无需 Vectorize |
| LLM | 同一 GLM API | 保持一致 |
| 知识库同步 | GitHub Actions / 手动脚本 | 定期从 RPi 推送快照 |

### 为什么不用向量数据库？

46 个 wiki 页面，向量维度 1024，总数据量 < 2MB。用 JSON 文件存储 + Python 内存加载 + 余弦相似度计算，检索耗时 < 50ms。不需要 Pinecone、Vectorize 等外部服务。

## 八、数据流

### 离线准备（一次性 + 知识库更新时重跑）

```
Obsidian Vault (SMB)
    │
    ▼ 读取 46 个 wiki markdown 文件
    │
    ▼ 过滤：排除 000_Journal/、001_Raw/、999_Template/
    │
    ▼ 分块：每页为一个 chunk（页面不大的不需要再切分）
    │
    ▼ 调用 Embedding API（bge-m3 / GLM embedding）
    │
    ▼ 生成 embeddings.json：
    │   [
    │     {"file": "002_Entities/傅颉 Freddie.md",
    │      "title": "傅颉 Freddie",
    │      "text": "完整页面内容...",
    │      "vector": [0.123, -0.456, ...]},
    │     ...
    │   ]
    │
    ▼ 部署到：
    ├── 树莓派：/home/pi/Projects/AskMe/data/embeddings.json
    └── 外网：推送到 GitHub 仓库 → Worker 构建时打包
```

### 在线查询（每次对话）

```
HR 输入问题
    │
    ▼ 前端 POST /api/chat {"message": "...", "history": [...]}
    │
    ▼ Worker/FastAPI 处理：
    │   1. 用户问题 → Embedding API → 问题向量
    │   2. 问题向量 vs embeddings.json 所有向量 → 余弦相似度
    │   3. 取 top-5 最相关页面
    │   4. 组装 prompt：
    │      ┌─────────────────────────────────┐
    │      │ SYSTEM: 你是傅颉的简历助手。      │
    │      │ 仅根据以下知识库内容回答。         │
    │      │ 如果知识库中没有相关信息，          │
    │      │ 明确说"我没有这方面的信息"。        │
    │      │ 不要编造任何内容。                 │
    │      │                                 │
    │      │ [知识库页面1]                     │
    │      │ [知识库页面2]                     │
    │      │ ...                              │
    │      │ [知识库页面5]                     │
    │      │                                 │
    │      │ HISTORY: (最近3轮对话)            │
    │      │ USER: {用户问题}                  │
    │      └─────────────────────────────────┘
    │   5. 调用 GLM API，流式返回
    │
    ▼ SSE (Server-Sent Events) 流式推送到前端
    │
    ▼ 前端逐字渲染（打字机效果）
```

### 节点A 数据实时更新流程

```
用户在 PC 上编辑 Obsidian Vault → SMB 自动同步
    │
    ▼ 触发方式（二选一）：
    │   a) 手动：调用 /api/reindex → 重新生成 embeddings.json
    │   b) 定时：cron 每6小时自动 reindex
    │
    ▼ embeddings.json 更新 → 下次对话使用新数据
```

### 节点B 知识库同步流程

```
树莓派 embeddings.json
    │
    ▼ git push 到 GitHub 仓库（手动或 cron）
    │
    ▼ Cloudflare Pages 自动构建部署
    │
    ▼ 同步延迟：最多数小时（可接受）
```

## 九、前端设计

### 手机端聊天界面

```
┌─────────────────────────┐
│  ← 傅颉 | Resume Bot     │  ← 顶栏，固定
├─────────────────────────┤
│                         │
│  👋 你好！我是傅颉的     │  ← 欢迎语（首次进入）
│  AI助手，可以问我关于    │
│  他的工作经历、项目经验  │
│  和技能背景。            │
│                         │
│          你在百融云创    │  ← 用户消息（右侧）
│          做了什么？      │
│                         │
│  🤖 在百融云创科技股份   │  ← AI回复（左侧，流式）
│  有限公司，傅颉担任...   │
│                         │
├─────────────────────────┤
│  [输入你的问题...    ] ➤│  ← 输入框，固定底部
└─────────────────────────┘
```

### 交互细节

- 首次进入显示欢迎语 + 3个推荐问题（可点击）
- 推荐问题示例："介绍一下工作经历"、"有哪些项目经验"、"AI相关的技能有哪些"
- 流式输出，逐字渲染
- 发送中禁用输入框，防止重复提交
- 无登录、无注册、无cookie
- 响应式：手机竖屏优先，平板/PC也兼容

### 兜底切换（前端逻辑）

```javascript
// 前端保存两个 API 端点
const API_PRIMARY = "https://media.logth.ink:18765";   // FRP → 树莓派
const API_FALLBACK = "https://askme.pages.dev";         // Cloudflare Pages

async function sendMessage(msg) {
  try {
    return await fetch(API_PRIMARY + "/api/chat", ...);
  } catch {
    // 节点A不可达，自动切换节点B
    return await fetch(API_FALLBACK + "/api/chat", ...);
  }
}
```

## 十四、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| LLM 幻觉 | 回答不准确 | system prompt 硬约束 + 仅用检索到的知识库内容 |
| 树莓派宕机 | 节点A不可用 | 前端自动切换到节点B |
| 家庭网络中断 | FRP 隧道断开 | 节点B 始终在线兜底 |
| 知识库过时 | 回答信息陈旧 | 定期 reindex + cron 自动化 |
| GLM API 不稳定 | 无法生成回答 | 多 LLM fallback（DeepSeek 备用） |
| FRP 无 HTTPS | 安全隐患 | 套 Cloudflare CDN 或 FRP 服务端配 TLS |
| embedding API 调用成本 | 费用增加 | 缓存问题 embedding，减少重复调用 |
